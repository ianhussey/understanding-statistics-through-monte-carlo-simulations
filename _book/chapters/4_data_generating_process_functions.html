In Chapter 3, you learned how to write functions. In this chapter, you’ll use that skill to build one of the core pieces of a simulation: a data-generating process (DGP) function.

Remember: a Monte Carlo simulation has five steps:

  1. Design an experiment
  2. Generate data
  3. Analyse data
  4. Do this many times
  5. Summarise results

It is easiest to talk about designing a simulation experiment after you know how to implement the other steps in a simulation. So this chapter is about Step 2: generate data. We’ll keep the “design” part simple for now (single choices for things like n and effect sizes).

Recall that in the last chapter you learned about how to write functions in R. In this chapter, we’re focusing specifically on writing functions that generate data and model data-generating processes.

4.2 What is a data-generating process?

A data-generating process (DGP) is a description of how your data come to exist.

  • In a real study, the world is the DGP (for example, the heights of humans in the population follow a normal distribution).
  • In a simulation, you decide what this data-generating process looks like, and then write code to generate data following this process.

A data-generating process is defined by parameters. These, put simply, define what is “true” in the world that we are simulating. For example, let’s imagine we want to simulate heights of a population between two groups (e.g., children and adults). In this case, we might decide that the true mean difference in heights between these two groups is 70cm. This means that the value for the parameter of “mean difference in height” is 70cm.

When we specify this DGP, we might then want to take random draws from it: this is equivalent to taking a sample from a population in a real study.

Remember that functions have three properties:

  1. They take inputs
  2. They do stuff to the inputs
  3. They return outputs

A DGP function’s properties are, then, more specifically:

  1. Take parameters as inputs
  2. Simulate data based on these parameters
  3. Return one simulated dataset based on these parameters.

4.3 Randomness and reproducibility

You might have heard people talking about “seeds” before when using R. Seeds, basically, are used to make sure that random output can be reproducible.

For example, look at the outputs of two calls to the rnorm() function below, which samples random values from a normal distribution with a mean of 0 and a standard deviation of 1:

Code 
output1 <- rnorm(n = 5, mean = 0, sd = 1)
output2 <- rnorm(n = 5, mean = 0, sd = 1)

output1 
[1]  0.5228723 -0.8453698  0.8424279  0.1633108 -1.4082423
Code 
output2
[1]  0.3417569  0.7267087  0.4921921 -0.2180675 -0.5273190

You can see that the outputs of the two calls are different. But sometimes we might not want this: for example, when we want other people to be able to reproduce our simulation results exactly. If we use truly random numbers, then every time we re-run the simulation we’ll get slightly different results. What we really want are pseudo-random numbers - numbers which are sampled randomly, but where the specific random sampling can be reconstructed and reproduced. This is exactly what seeds are for: if you set a seed to a specific value, then all calls to functions which use randomisation will reproduce their random samples in future runs.

Put simply: if you start from the same seed, you get the same sequence of “random” values. Check out the example below, now using set.seed():

Code 
set.seed(123)

output1 <- rnorm(n = 5, mean = 0, sd = 1)
output2 <- rnorm(n = 5, mean = 0, sd = 1)

output1 
[1] -0.56047565 -0.23017749  1.55870831  0.07050839  0.12928774
Code 
output2
[1]  1.7150650  0.4609162 -1.2650612 -0.6868529 -0.4456620

In your own work, you’ll often set a seed once at the top of a script so you (and others) can reproduce your results.

A common mistake: setting the seed inside your generator

Don’t put set.seed() inside a DGP function. If you do, every call will restart the random number generator in the same place, and you will keep generating the same dataset. This becomes a problem when we get to Step 4 (repeat many times), since you just end up repeating the exact same process every time.

Code 
bad_generate <- function() {
  set.seed(1)
  rnorm(3)
}

bad_generate()
[1] -0.6264538  0.1836433 -0.8356286
Code 
bad_generate()
[1] -0.6264538  0.1836433 -0.8356286
Code 
set.seed(1)
good_generate <- function() {
  rnorm(3)
}

good_generate()
[1] -0.6264538  0.1836433 -0.8356286
Code 
good_generate()
[1]  1.5952808  0.3295078 -0.8204684

4.4 Building our first data-generating function: simulating two-group mean difference

One of the simplest examples that can be simulated - and the example that we’re going to use frequently in the next several chapters - is simulating two groups with a known difference in means.

The most classic version of this already classic example is to simulate a difference in means in a “control vs intervention” study with a continuous outcome. From your introductory statistics knowledge, you probably recognise that such a design produces the exact type of data we might analyse with an independent-samples t-test.

Recall that a good data-generating function has three qualities: inputs, do stuff, and outputs.

Let’s think about our inputs first. What types of things might vary in such a design? The obvious choice is the means in the control and intervention groups. We could also vary the standard deviation (SD) associated with the means. And we could also vary the number of participants we have in each condition. This means we have as inputs:

  • n_per_condition: sample size per group
  • mean_control: mean in the control group
  • mean_intervention: mean in the intervention group
  • sd: within-group standard deviation

We will use R’s rnorm() function to generate the data for each group. rnorm() takes three arguments: n (the sample size), mean (the mean value), and sd (the standard deviation). rnorm() will then give us n random draws from a normal distribution with a mean of mean and an SD of sd.

Let’s remember how to write a function: write out the desired process, then wrap it in a function.

4.4.1 Step 1: generate one dataset (no function yet)

Start by getting the data generation working with hard-coded values. Here’s what our pseudocode for this looks like:

Code 

BEGIN

1) set seed for reproducibility
2) Define simulation parameters for:
  - n_per_condition
  - mean_control 
  - mean_intervention   
  - sd                  
3) Generate data for control group
data_control is a TABLE with n_per_condition rows:
  - id = n_per_condition value
  - condition = control
  - score = random value from normal distribution with mean = mean_control and sd = sd
4) Generate data for intervention group (non-overlapping IDs)
data_intervention is a TABLE with n_per_condition rows:
  - id = n_per_condition value, not overlapping with id from data_control
  - condition = intervention
  - score = random value from normal distribution with mean = mean_intervention and sd = sd
5) Combine the two groups into one dataset
6) Output the dataset
END

4.4.2 Step 2: wrap it into a DGP function

Now let’s turn our working code into a reusable function (as we did previously in Chapter 3).

Code 
generate_data_two_group <- function(n_per_condition,
                                    mean_control,
                                    mean_intervention,
                                    sd) {

  data_control <- tibble(
    id = 1:n_per_condition,
    condition = "control",
    score = rnorm(n_per_condition, mean = mean_control, sd = sd)
  )

  data_intervention <- tibble(
    id = (n_per_condition + 1):(2 * n_per_condition),
    condition = "intervention",
    score = rnorm(n_per_condition, mean = mean_intervention, sd = sd)
  )

  dat <- bind_rows(data_control, data_intervention) |>
    as.tibble()

  return(dat)
}

4.4.3 Why return a tibble?

In the next chapter, you’ll write analysis functions that take a dataset and return a result (a p-value, an estimated effect, a confidence interval, etc). When we write these analysis functions, we will have certain expectations about the output of our data-generating functions (e.g., that they will conform to principles of tidy data).

In order to ensure that every data-generating function returns a tidy dataset with predictable column names, it is good practice to make sure that every DGP function returns a tibble as output. Tibbles are like dataframes, but with more consistent behavior that is useful for analysis. For example, if we use select() and select a single column from a dataframe, this is automatically treated by R as a vector, which might break some of our analysis functions down the line (which expect tibbles). select()ing one column from a tibble, by contrast, still returns a tibble object.

Importantly, the outputs of our DGP functions should usually consist of the following information:

  • one row per observation
  • an id column
  • relevant grouping variables
  • one column per variable you’ll analyse

4.5 A quick checklist for DGP functions

When you write your own generate_data_*() functions, aim for these defaults:

  • Put all “design choices” in the arguments (sample sizes, effect sizes, standard deviations, etc.).
  • Return one dataset per function call.
  • Avoid side effects (don’t call set.seed() inside the functionmdon’t write files, don’t make plots).
  • Use clear, consistent column names (which helps both for reading code and for running analysis functions later).

4.6 Common data-generating functions

In practice, many of the data-generating processes that we want to simulate can be modelled using some useful functions in R. You have already seen one example of this used above: namely, rnorm(). In this section, we’ll go through a small number of additional data-generating functions that you will likely find useful in your future simulations.

4.6.1 Simulating binary outcomes with rbinom()

rbinom() is used to model binomial distributions. This might sound a little offputting, but it’s actually very simple: binomial distributions refer to cases where there are only two possible answers. Some examples of such situations are heads/tails in coin-flips, correct/incorrect in testing, dropped out/finished in an intervention study, and many others.

Let’s look at how to use rbinom(). It takes three arguments: n, size, and prob. n refers to the number of samples (e.g., how many total rounds of coin-flips are done); size refers to the number of observations per sample (e.g., 10 coin-flips are done for every round of coin-flips collected); and prob refers to the “probability of success” (e.g., in coin-flips both outcomes are equally likely, so the probability of “success” (defined either as getting heads or getting tails) is 0.5).

Let’s look at a simple example: we want to simulate 10 rounds of flipping a single coin, assuming a probability of success of 0.5 for that coin.

Code 
set.seed(2)
rbinom(n = 10, size = 1, prob = 0.5)
 [1] 0 1 1 0 1 1 0 1 0 1

Let’s expand this into our two-group example from before. Previously, we used rnorm() to simulate values for a continuous outcome; now we can simulate binary values (e.g., getting a test question correct, scored as 1, or incorrect, scored as 0) with differences in the probability of correct responses between the two groups. This looks like:

Code 
generate_data_two_group_binary <- function(n_per_condition,
                                           prob_control,
                                           prob_intervention) {

  data_control <- tibble(
    id = 1:n_per_condition,
    condition = "control",
    score = rbinom(n_per_condition, size = 1, prob = prob_control)
  )

  data_intervention <- tibble(
    id = (n_per_condition + 1):(2 * n_per_condition),
    condition = "intervention",
    score = rbinom(n_per_condition, size = 1, prob = prob_intervention)
  )

  dat <- bind_rows(data_control, data_intervention) |>
    as_tibble()

  return(dat)
}

We can check the observed proportions to make sure our functions is working correctly. Specifically, if we simulate with n_per_condition = 30, prob_control = 0.25, and prob_intervention = 0.75, then we should see these proportions in the estimated mean scores of each group:

Code 
set.seed(2)

dat_bin <- generate_data_two_group_binary(
  n_per_condition = 30,
  prob_control = 0.25,
  prob_intervention = 0.75
)

dat_bin |>
  group_by(condition) |>
  summarise(estimated_mean_score = mean(score), .groups = "drop")
# A tibble: 2 × 2
  condition    estimated_mean_score
  <chr>                       <dbl>
1 control                     0.267
2 intervention                0.6

Using rbinom() is crucial for simulating binary outcome DGPs. Later, your analysis function might be use a logistic regression to analyse these binary data. But note that, even though the specific DGP function has changed, the structure of the generate_data_two_group_binary function looks the same as generate_data_two_group from before: take inputs -> do stuff -> output a tibble.

4.6.2 Correlated data: repeated measures

Psychology data often include repeated measurements on the same person. For example, we might measure a person’s score on a scale before and after they receive an intervention. In principle, we could just simulate these observations using two different rnorm() calls, like this:

Code 
generate_data_two_timepoints <- function(n_per_timepoint,
                                    mean_score_t1,
                                    mean_score_t2,
                                    sd) {

  dat <- tibble(
    id = 1:n_per_timepoint,
    score_t1 = rnorm(n_per_timepoint, mean = mean_score_t1, sd = sd),
    score_t2 = rnorm(n_per_timepoint, mean = mean_score_t2, sd = sd)
  )

  return(dat)
}

But let’s run this function and look at the correlation between these observations:

Code

                                                   mean_score_t1 = 0,
                                                   mean_score_t2 = 0.5,
                                                   sd = 1)

broom::tidy(
  cor.test(two_timepoint_data$score_t1,
           two_timepoint_data$score_t2)
)
# A tibble: 1 × 8
  estimate statistic p.value parameter conf.low conf.high method     alternative
     <dbl>     <dbl>   <dbl>     <int>    <dbl>     <dbl> <chr>      <chr>      
1  -0.0391    -0.271   0.788        48   -0.314     0.242 Pearson's… two.sided

The scores from timepoint 1 (T1) and timepoint 2 (T2) are uncorrelated. But in real life, this rarely happens: two observations from the same participant across time tend to be correlated, even when they are influenced by an intervention.

So, how do we simulate correlated variables?

One simple way to do this is to draw from what is known as a multivariate normal distribution. This is a distribution that lets us specify both the means and SDs of variables, as well as the correlation between them. The function mvrnorm() from the {MASS} library does this very well.

Let’s simulate a design where we have a pre-intervention score variable mean-pre, and a post-ntervention score variable mean-post, and a correlation between these two variables cor_pre_post.

To use a multivariate normal distribution, we need to specify the covariance matrix between the variables. This is just a matrix that specifies how related each variable is to each other. Since we have two variables here, we need to create a 2x2 matrix.

You can think of a covariance matrix as an unstandardised correlation matrix. The correlation between a variable and itself is always 1. The covariance between a variable and itself is always the square of its standard deviation, namely sd^2.

When the SD is the same at both time points, the covariance between two variables is the correlation between them multiplied by sd^2, AKA cor_pre_post * sd^2. When we use the mvrnorm() function, we supply this covariance matrix to it as the parameter Sigma.

Our simple multivariate DGP function is as follows:

Code 
generate_data_pre_post <- function(n,
                                   mean_pre,
                                   mean_post,
                                   sd,
                                   cor_pre_post) {

  # construct the covariance matrix
  covariance_matrix <- matrix(
    c(sd^2, cor_pre_post * sd^2,
      cor_pre_post * sd^2, sd^2),
    nrow = 2,
    byrow = TRUE
  )

  # simulate from the multivariate normal distribution
  raw_dat <- MASS::mvrnorm(
    n = n,
    mu = c(mean_pre, mean_post),
    Sigma = covariance_matrix
  )


  dat <- as_tibble(raw_dat) |>
    setNames(c("score_pre", "score_post")) |>
    mutate(id = 1:n) |> # create an identifier
    relocate(id)

  return(dat)
}

4.6.3 Many variables at once: faux::rnorm_multi()

{MASS}’s mvrnorm() function is handy for generating multiple correlated variables. However, you may have noticed a couple of things that are not ideal about it. First, we need to manually construct the covariance matrix, which requires a little bit of fiddling and is generally a bit of a pain - especially if we want to simulate many variables, it can become quite a lot of work. Second, mvrnorm() does not give tidy output, meaning we have to do extra data cleaning and processing.

Fortunately, there exists a much handier function for simulating multivariate normal data: rnorm_multi() from the {faux} R package. We can use rnorm_multi() to simulate many correlated variables at once. This is especially useful for tasks like generating many items of a scale which should be correlated.

If you don’t have it installed yet, install it once:

Code 
install.packages("faux")

rnorm_multi() can take a few different arguments. They are:

  • n = the sample size per variable
  • vars = the number of variables to be generated
  • mu = the means of the variables
  • sd = the SDs of the variables
  • r = the correlation between each of the variables
  • varnames = the desired names of the variables
  • empirical = whether the correlations and means should exactly match the input values or not
  • as.matrix = whether the output should be returned as a matrix (TRUE) or not (FALSE)

Obviously, we want our output as a tibble, not a matrix, so we will set as.matrix to FALSE. This is actually the default value, so we can leave it alone.

Let’s use rnorm_multi() to generate 5 variables, each with 200 observations n, each with a mean mu of 0, a sd sd of 1, and a correlation r of 0.3.

Code 
set.seed(1)

data_simulated_items <- faux::rnorm_multi(
  n = 200,
  vars = 5,
  mu = 0,
  sd = 1,
  r = 0.3,
  varnames = c("variable1", "variable2", "variable3", "variable4", "variable5")
) |>
  as_tibble()

A full DGP might simulate these items, and then compute a total score based on these individual items:

Code 
generate_data_scale_score <- function(n, 
                                mean, 
                                sd, 
                                n_items,
                                r) {

  items <- faux::rnorm_multi(
    n = n,
    vars = n_items,
    mu = mean,
    sd = sd,
    r = r,
    varnames = paste0("variable", 1:n_items)
  ) |>
    as_tibble()

  dat <- tibble(id = 1:n) |>
    bind_cols(items) |>
    mutate(across(contains("variable"), ~mean(.)))

  return(dat)
}

# Data-generating process functions

In Chapter 3, you learned how to write functions. In this chapter, you’ll use that skill to build one of the core pieces of a simulation: **a data-generating process (DGP) function**.

Remember: a Monte Carlo simulation has five steps:

1. Design an experiment  
2. Generate data  
3. Analyse data  
4. Do this many times  
5. Summarise results  

It is easiest to talk about designing a simulation experiment after you know how to implement the other steps in a simulation. So this chapter is about Step 2: **generate data**. We’ll keep the “design” part simple for now (single choices for things like `n` and effect sizes).

Recall that in the last chapter you learned about how to write functions in R. In this chapter, we're focusing specifically on writing functions that generate data and model data-generating processes.

## Dependencies

```{r}
library(dplyr)
library(tidyr)
```

## What is a data-generating process?

A **data-generating process (DGP)** is a description of how your data come to exist.

- In a real study, the world is the DGP (for example, the heights of humans in the population follow a normal distribution).
- In a simulation, *you* decide what this data-generating process looks like, and then write code to generate data following this process.

A data-generating process is defined by **parameters**. These, put simply, define what is "true" in the world that we are simulating. For example, let's imagine we want to simulate heights of a population between two groups (e.g., children and adults). In this case, we might decide that the true mean difference in heights between these two groups is 70cm. This means that the value for the parameter of "mean difference in height" is 70cm. 

When we specify this DGP, we might then want to take **random draws** from it: this is equivalent to taking a sample from a population in a real study. 

Remember that functions have three properties:

1. They take inputs
2. They do stuff to the inputs
3. They return outputs

A DGP function's properties are, then, more specifically:

1. Take parameters as inputs
2. Simulate data based on these parameters
3. Return **one** simulated dataset based on these parameters.

## Randomness and reproducibility

You might have heard people talking about "seeds" before when using R. Seeds, basically, are used to make sure that random output can be reproducible. 

For example, look at the outputs of two calls to the `rnorm()` function below, which samples random values from a normal distribution with a mean of 0 and a standard deviation of 1:

```{r}
output1 <- rnorm(n = 5, mean = 0, sd = 1)
output2 <- rnorm(n = 5, mean = 0, sd = 1)

output1 
output2

```

You can see that the outputs of the two calls are different. But sometimes we might not want this: for example, when we want other people to be able to reproduce our simulation results exactly. If we use truly random numbers, then every time we re-run the simulation we'll get slightly different results. What we really want are *pseudo*-random numbers - numbers which are sampled randomly, but where the specific random sampling can be reconstructed and reproduced. This is exactly what seeds are for: if you set a seed to a specific value, then all calls to functions which use randomisation will reproduce their random samples in future runs. 

Put simply: if you start from the same seed, you get the same sequence of "random" values. Check out the example below, now using `set.seed()`:

```{r}
set.seed(123)

output1 <- rnorm(n = 5, mean = 0, sd = 1)
output2 <- rnorm(n = 5, mean = 0, sd = 1)

output1 
output2

```

In your own work, you’ll often set a seed once at the top of a script so you (and others) can reproduce your results.

::: {.callout-warning collapse="true" title="A common mistake: setting the seed inside your generator"}
Don’t put `set.seed()` *inside* a DGP function. If you do, every call will restart the random number generator in the same place, and you will keep generating the same dataset. This becomes a problem when we get to Step 4 (repeat many times), since you just end up repeating the exact same process every time.

```{r}
bad_generate <- function() {
  set.seed(1)
  rnorm(3)
}

bad_generate()
bad_generate()

```

```{r}
set.seed(1)
good_generate <- function() {
  rnorm(3)
}

good_generate()
good_generate()

```

## Building our first data-generating function: simulating two-group mean difference

One of the simplest examples that can be simulated - and the example that we're going to use frequently in the next several chapters - is simulating two groups with a known difference in means. 

The most classic version of this already classic example is to simulate a difference in means in a "control vs intervention" study with a continuous outcome. From your introductory statistics knowledge, you probably recognise that such a design produces the exact type of data we might analyse with an independent-samples *t*-test.

Recall that a good data-generating function has three qualities: inputs, do stuff, and outputs. 

Let's think about our inputs first. What types of things might vary in such a design? The obvious choice is the means in the control and intervention groups. We could also vary the standard deviation (SD) associated with the means. And we could also vary the number of participants we have in each condition. This means we have as inputs:

- `n_per_condition`: sample size per group
- `mean_control`: mean in the control group
- `mean_intervention`: mean in the intervention group
- `sd`: within-group standard deviation

We will use R's `rnorm()` function to generate the data for each group. `rnorm()` takes three arguments: `n` (the sample size), `mean` (the mean value), and `sd` (the standard deviation). `rnorm()` will then give us `n` random draws from a normal distribution with a mean of `mean` and an SD of `sd`.

Let's remember how to write a function: write out the desired process, then wrap it in a function.

### Step 1: generate one dataset (no function yet)

Start by getting the data generation working with hard-coded values. Here's what our pseudocode for this looks like:

```{default}
BEGIN

1) set seed for reproducibility
2) Define simulation parameters for:
  - n_per_condition
  - mean_control 
  - mean_intervention   
  - sd                  
3) Generate data for control group
data_control is a TABLE with n_per_condition rows:
  - id = n_per_condition value
  - condition = control
  - score = random value from normal distribution with mean = mean_control and sd = sd
4) Generate data for intervention group (non-overlapping IDs)
data_intervention is a TABLE with n_per_condition rows:
  - id = n_per_condition value, not overlapping with id from data_control
  - condition = intervention
  - score = random value from normal distribution with mean = mean_intervention and sd = sd
5) Combine the two groups into one dataset
6) Output the dataset
END

```

If all is in order, `n` should be 50 for each group; the mean of `score` should be roughly 0 for the control condition and roughly 0.5 for the intervention condition; and SDs should be roughly 1 for both groups.

### Why return a tibble?

In the next chapter, you’ll write **analysis functions** that take a dataset and return a result (a *p*-value, an estimated effect, a confidence interval, etc). When we write these analysis functions, we will have certain expectations about the output of our data-generating functions (e.g., that they will conform to principles of tidy data).

In order to ensure that every data-generating function returns a tidy dataset with predictable column names, it is good practice to make sure that every DGP function returns a tibble as output. Tibbles are like dataframes, but with more consistent behavior that is useful for analysis. For example, if we use `select()` and select a single column from a dataframe, this is automatically treated by R as a vector, which might break some of our analysis functions down the line (which expect tibbles). `select()`ing one column from a tibble, by contrast, still returns a tibble object.

Importantly, the outputs of our DGP functions should usually consist of the following information:

- one row per observation
- an `id` column
- relevant grouping variables
- one column per variable you’ll analyse


## A quick checklist for DGP functions

When you write your own `generate_data_*()` functions, aim for these defaults:

- Put all “design choices” in the arguments (sample sizes, effect sizes, standard deviations, etc.).
- Return **one** dataset per function call.
- Avoid side effects (don’t call `set.seed()` inside the functionmdon’t write files, don’t make plots).
- Use clear, consistent column names (which helps both for reading code and for running analysis functions later).

## Common data-generating functions

In practice, many of the data-generating processes that we want to simulate can be modelled using some useful functions in R. You have already seen one example of this used above: namely, `rnorm()`. In this section, we'll go through a small number of additional data-generating functions that you will likely find useful in your future simulations.

### Simulating binary outcomes with `rbinom()`

`rbinom()` is used to model binomial distributions. This might sound a little offputting, but it's actually very simple: binomial distributions refer to cases where there are only two possible answers. Some examples of such situations are heads/tails in coin-flips, correct/incorrect in testing, dropped out/finished in an intervention study, and many others. 

Let's look at how to use `rbinom()`. It takes three arguments: `n`, `size`, and `prob`. `n` refers to the number of samples (e.g., how many total rounds of coin-flips are done); `size` refers to the number of observations per sample (e.g., 10 coin-flips are done for every round of coin-flips collected); and `prob` refers to the "probability of success" (e.g., in coin-flips both outcomes are equally likely, so the probability of "success" (defined either as getting heads or getting tails) is 0.5).

Let's look at a simple example: we want to simulate 10 rounds of flipping a single coin, assuming a probability of success of 0.5 for that coin. 

```{r}
set.seed(2)
rbinom(n = 10, size = 1, prob = 0.5)

```

Let's expand this into our two-group example from before. Previously, we used `rnorm()` to simulate values for a continuous outcome; now we can simulate binary values (e.g., getting a test question correct, scored as 1, or incorrect, scored as 0) with differences in the probability of correct responses between the two groups. This looks like:

```{r}
generate_data_two_group_binary <- function(n_per_condition,
                                           prob_control,
                                           prob_intervention) {

  data_control <- tibble(
    id = 1:n_per_condition,
    condition = "control",
    score = rbinom(n_per_condition, size = 1, prob = prob_control)
  )

  data_intervention <- tibble(
    id = (n_per_condition + 1):(2 * n_per_condition),
    condition = "intervention",
    score = rbinom(n_per_condition, size = 1, prob = prob_intervention)
  )

  dat <- bind_rows(data_control, data_intervention) |>
    as_tibble()

  return(dat)
}

```

We can check the observed proportions to make sure our functions is working correctly. Specifically, if we simulate with `n_per_condition` = 30, `prob_control` = 0.25, and `prob_intervention` = 0.75, then we should see these proportions in the estimated mean scores of each group:

```{r}
set.seed(2)

dat_bin <- generate_data_two_group_binary(
  n_per_condition = 30,
  prob_control = 0.25,
  prob_intervention = 0.75
)

dat_bin |>
  group_by(condition) |>
  summarise(estimated_mean_score = mean(score), .groups = "drop")

```

Using `rbinom()` is crucial for simulating binary outcome DGPs. Later, your analysis function might be use a logistic regression to analyse these binary data. But note that, even though the specific DGP function has changed, the *structure* of the `generate_data_two_group_binary` function looks the same as `generate_data_two_group` from before: take inputs -> do stuff -> output a tibble.

### Correlated data: repeated measures

Psychology data often include repeated measurements on the same person. For example, we might measure a person's score on a scale before and after they receive an intervention. In principle, we could just simulate these observations using two different `rnorm()` calls, like this:

```{r}
generate_data_two_timepoints <- function(n_per_timepoint,
                                    mean_score_t1,
                                    mean_score_t2,
                                    sd) {

  dat <- tibble(
    id = 1:n_per_timepoint,
    score_t1 = rnorm(n_per_timepoint, mean = mean_score_t1, sd = sd),
    score_t2 = rnorm(n_per_timepoint, mean = mean_score_t2, sd = sd)
  )

  return(dat)
}


```

But let's run this function and look at the correlation between these observations:

```{r}
set.seed(1)

two_timepoint_data <- generate_data_two_timepoints(n_per_timepoint = 50,
                                                   mean_score_t1 = 0,
                                                   mean_score_t2 = 0.5,
                                                   sd = 1)

broom::tidy(
  cor.test(two_timepoint_data$score_t1,
           two_timepoint_data$score_t2)
)

```

The scores from timepoint 1 (T1) and timepoint 2 (T2) are uncorrelated. But in real life, this rarely happens: two observations from the same participant across time tend to be correlated, even when they are influenced by an intervention. 

So, how do we simulate *correlated* variables?

One simple way to do this is to draw from what is known as a **multivariate normal distribution**. This is a distribution that lets us specify both the means and SDs of variables, as well as the correlation between them. The function `mvrnorm()` from the {MASS} library does this very well.

Let's simulate a design where we have a pre-intervention score variable `mean-pre`, and a post-ntervention score variable `mean-post`, and a correlation between these two variables `cor_pre_post`. 

To use a multivariate normal distribution, we need to specify the *covariance matrix* between the variables. This is just a matrix that specifies how related each variable is to each other. Since we have two variables here, we need to create a 2x2 matrix. 

You can think of a covariance matrix as an unstandardised correlation matrix. The correlation between a variable and itself is always 1. The covariance between a variable and itself is always the square of its standard deviation, namely `sd^2`. 

When the SD is the same at both time points, the covariance between two variables is the correlation between them multiplied by `sd^2`, AKA `cor_pre_post * sd^2`. When we use the `mvrnorm()` function, we supply this covariance matrix to it as the parameter `Sigma`. 

Our simple multivariate DGP function is as follows:

```{r}
generate_data_pre_post <- function(n,
                                   mean_pre,
                                   mean_post,
                                   sd,
                                   cor_pre_post) {

  # construct the covariance matrix
  covariance_matrix <- matrix(
    c(sd^2, cor_pre_post * sd^2,
      cor_pre_post * sd^2, sd^2),
    nrow = 2,
    byrow = TRUE
  )

  # simulate from the multivariate normal distribution
  raw_dat <- MASS::mvrnorm(
    n = n,
    mu = c(mean_pre, mean_post),
    Sigma = covariance_matrix
  )


  dat <- as_tibble(raw_dat) |>
    setNames(c("score_pre", "score_post")) |>
    mutate(id = 1:n) |> # create an identifier
    relocate(id)

  return(dat)
}

```

And here's what the output of this looks like:

```{r}
set.seed(1)

data_pre_post <- generate_data_pre_post(
  n = 50,
  mean_pre = 0,
  mean_post = 0.5,
  sd = 1,
  cor_pre_post = 0.7
)

data_pre_post

```

Once again, we can sanity-check our code to make sure we see both means and between-variable correlations like we expect:

```{r}

  summarise(mean_pre = mean(score_pre),
            mean_post = mean(score_post),
```

### Many variables at once: `faux::rnorm_multi()`

{MASS}'s `mvrnorm()` function is handy for generating multiple correlated variables. However, you may have noticed a couple of things that are not ideal about it. First, we need to manually construct the covariance matrix, which requires a little bit of fiddling and is generally a bit of a pain - especially if we want to simulate many variables, it can become quite a lot of work. Second, `mvrnorm()` does not give tidy output, meaning we have to do extra data cleaning and processing. 

Fortunately, there exists a much handier function for simulating multivariate normal data: `rnorm_multi()` from the {faux} R package. We can use `rnorm_multi()` to simulate many correlated variables at once. This is especially useful for tasks like generating many items of a scale which should be correlated.

If you don’t have it installed yet, install it once:

```{r}
#| eval: false

install.packages("faux")

```

`rnorm_multi()` can take a few different arguments. They are:

- `n` = the sample size per variable
- `vars` = the number of variables to be generated
- `mu` = the means of the variables
- `sd` = the SDs of the variables
- `r` = the correlation between each of the variables
- `varnames` = the desired names of the variables
- `empirical` = whether the correlations and means should exactly match the input values or not
- `as.matrix` = whether the output should be returned as a matrix (TRUE) or not (FALSE)

Obviously, we want our output as a tibble, not a matrix, so we will set `as.matrix` to FALSE. This is actually the default value, so we can leave it alone.

Let's use `rnorm_multi()` to generate 5 variables, each with 200 observations `n`, each with a mean `mu` of 0, a sd `sd` of 1, and a correlation `r` of 0.3. 

```{r}

  sd = 1,

) |>
  as_tibble()

```

Let's look at the correlations between the simulated items:

Setting `empirical = TRUE` can make the simulation take longer to run, but it can also provide us with data that more precisely adheres to our specified parameter values. Although we usually do not want this in a simulation where we do this process many times.

A full DGP might simulate these items, and then compute a total score based on these individual items:

```{r}
generate_data_scale_score <- function(n, 
                                mean, 
                                sd, 
                                n_items,
                                r) {

  items <- faux::rnorm_multi(
    n = n,
    vars = n_items,
    mu = mean,
    sd = sd,
    r = r,
    varnames = paste0("variable", 1:n_items)
  ) |>
    as_tibble()

  dat <- tibble(id = 1:n) |>
    bind_cols(items) |>
    mutate(across(contains("variable"), ~mean(.)))

  return(dat)
}

```

Let's run this function where we assume 100 participants completed 10 items, each item having a mean of 0, SD of 1, and a correlation with the other items of 0.5:

```{r}
generate_data_scale_score(n = 100, 
                          mean = 0, 
                          sd = 1, 
                          n_items = 10,
                          r = 0.5)


```

What is returned, as we specified in our function above, is a single score for each participant, representing the mean of the 10 item scores for that participant. But we know that these scores are produced based on 10 items with a mean of 0, SD of 1, and correlation of 0.5 among them.
